# Phase 2A — TPC-H Workload Integration

> Advanced reference for the TPC-H benchmark layer of the
> **AutoML-Powered Learned Query Optimizer**. Read this together with
> `docs/PHASE1_EXPLAINED.md` — Phase 2A reuses everything Phase 1
> built and extends it to a research-grade workload.

---

## 0. Why TPC-H?

Phase 1's synthetic `customers`/`orders` schema is great for sanity
checks, but it's *too clean*: uniform distributions, no skew, only two
tables, only a handful of plan shapes. A learned cost model trained on
that workload would overfit to "easy" plans and have no idea how to
score a real analytic query.

**TPC-H** (Transaction Processing Performance Council, Decision Support
Benchmark "H") is the industry-standard analytical workload:

- **8 tables** with realistic referential structure (snowflake-ish).
- **22 queries** that exercise every major operator: filtered scans,
  inner/outer/semi/anti joins, correlated and uncorrelated subqueries,
  group-bys, sorts, top-k, CASE-based pivots.
- **Skewed, parameterised data** generated by a spec-compliant
  generator — not uniform `g % N` noise.
- Plans you collect here are directly comparable to every TPC-H paper
  ever published.

Phase 2A's job: get TPC-H into PostgreSQL on Windows with one
`pip install`, run all 22 queries, and capture **multiple plan
variants per query** so we get a much richer dataset for Phase 2B
(feature extraction & cost regression).

## 2. End-to-end usage

```powershell
# 1. install (adds DuckDB)
pip install -r requirements.txt

# 2. generate TPC-H data + load into PostgreSQL
python scripts/setup_tpch.py            # SF=1 (~1 GB; ~6 M lineitem rows)
# faster on a laptop:
python scripts/setup_tpch.py --sf 0.1   # SF=0.1 (~100 MB)

# 3. collect plans (22 queries × 4 optimizer variants = 88 plans)
python scripts/collect_tpch_plans.py
```

Outputs:

- `data/tpch/raw_data/*.csv` — generated TPC-H data (gitignored).
- `data/tpch/plans/{qid}__{variant}__{hash}.json` — one plan per
  (query, variant).
- `data/tpch/plans/_index.jsonl` — one summary line per plan,
  pandas-friendly.

## 3. `db/tpch_schema.sql` — line-by-line

### 3.1 Drop order matters

```sql
DROP TABLE IF EXISTS lineitem CASCADE;
DROP TABLE IF EXISTS orders   CASCADE;
...
DROP TABLE IF EXISTS region   CASCADE;
```

TPC-H has a strict FK chain:

```
region ← nation ← supplier
                   ↑
            partsupp ← part
                   ↑
            lineitem → orders → customer → nation
```

We drop **leaves first** so PostgreSQL doesn't complain about FK
dependents. `CASCADE` is defensive — works even if you later add
views or extra constraints.

### 3.2 Types follow the TPC-H spec exactly

| Spec type    | PostgreSQL mapping  | Why |
|--------------|---------------------|-----|
| `IDENTIFIER` | `INTEGER`           | 32-bit fits up to SF~100; spec-compliant up to SF1000 with `BIGINT`. |
| `DECIMAL`    | `DECIMAL(15,2)`     | Exact decimals — required for `SUM(price*(1-disc)*(1+tax))` to match published reference results bit-for-bit. |
| `DATE`       | `DATE`              | Native, supports `INTERVAL` arithmetic. |
| `CHAR(N)`    | `CHAR(N)`           | Fixed-width with trailing space padding — matters for `l_returnflag = 'R'` style predicates. |
| `VARCHAR(N)` | `VARCHAR(N)`        | Variable length. |

> **Why `CHAR(1)` for `l_returnflag` and not `BOOLEAN`?** Because
> TPC-H's `l_returnflag` has three values (`R`, `A`, `N`). Mapping to
> bool would silently lose information. Faithfulness to the spec is
> non-negotiable here.

### 3.3 Indexes — research-grade defaults

```sql
CREATE INDEX idx_lineitem_shipdate     ON lineitem(l_shipdate);
CREATE INDEX idx_lineitem_orderkey     ON lineitem(l_orderkey);
CREATE INDEX idx_lineitem_partkey      ON lineitem(l_partkey);
CREATE INDEX idx_lineitem_suppkey      ON lineitem(l_suppkey);
CREATE INDEX idx_lineitem_part_supp    ON lineitem(l_partkey, l_suppkey);
CREATE INDEX idx_orders_orderdate      ON orders(o_orderdate);
CREATE INDEX idx_orders_custkey        ON orders(o_custkey);
CREATE INDEX idx_customer_nationkey    ON customer(c_nationkey);
CREATE INDEX idx_supplier_nationkey    ON supplier(s_nationkey);
CREATE INDEX idx_partsupp_suppkey      ON partsupp(ps_suppkey);
CREATE INDEX idx_nation_regionkey      ON nation(n_regionkey);
```

These are the **conventional TPC-H indexes** found in most academic
papers and in PG-tune-for-OLAP guides. They are deliberately
*incomplete* — we leave many filtering columns unindexed
(`p_brand`, `c_mktsegment`, `s_comment`, ...) so the planner has to
make real choices between Seq Scan, Bitmap Scan, and Index Scan.

If we indexed *everything*, every query would always pick an Index
Scan and our dataset would be boring.

## 4. `db/tpch_queries.sql` — design notes

### 4.1 Marker format

```sql
-- @QUERY: q03 | tag: join+agg+sort+limit
SELECT ...
;
```

The collector parses this with a single regex:

```python
HEADER_RE = re.compile(
    r"^--\s*@QUERY:\s*(?P<id>\S+)\s*\|\s*tag:\s*(?P<tag>.+?)\s*$",
    re.MULTILINE,
)
```

Why a marker convention instead of one-query-per-file?

- **Diffable.** Editing a parameter on Q14 is one line in one file.
- **Atomic.** "The workload" is a single artifact you can hash and
  version.
- **Extensible.** You can drop in custom queries with their own
  `@QUERY` headers and the collector picks them up with no code
  changes.

### 4.2 Parameter values are the TPC-H *reference* set

Every query uses the standard reference parameters from the TPC-H
specification (e.g. Q5 → `REGION = 'ASIA'` & year `1994`, Q21 →
`NATION = 'SAUDI ARABIA'`). These are the same constants used by
DuckDB, Snowflake's tpch-benchmark, and almost every published
TPC-H result. So:

- Your plans are directly comparable to others'.
- Re-runs are byte-stable.
- Anyone debugging "why is my Q9 plan different from yours?" can
  start from a common baseline.

### 4.3 Postgres-specific tweaks

A few queries needed mechanical changes to be legal SQL in PostgreSQL
while preserving the original semantics:

| Query | Original spec       | What we do                                                |
|-------|---------------------|-----------------------------------------------------------|
| Q1    | `date '1998-12-01' - 90 days` (DB2 syntax) | `DATE '1998-12-01' - INTERVAL '90' DAY`            |
| Q15   | Creates `VIEW revenue0` | Inline `WITH revenue0 AS (...)` CTE — same plan, no DDL |
| Q4/Q7/Q8/... | `date '...' + 1 year` | `DATE '...' + INTERVAL '1' YEAR`                  |
| Q22   | `substring(c_phone, 1, 2)` | `SUBSTRING(c_phone FROM 1 FOR 2)` (SQL-standard form) |

These are functionally identical to the spec — they keep all join
graphs, predicates, and aggregations bit-identical.

### 4.4 The 22 queries grouped by plan shape

| Plan shape         | Queries                  |
|--------------------|--------------------------|
| Pure aggregate     | Q1, Q6                   |
| Filter-heavy       | Q1, Q6, Q14, Q19         |
| 3-way join         | Q3, Q12                  |
| 4-way+ join        | Q5, Q7, Q8, Q9, Q21      |
| Correlated subq.   | Q2, Q17, Q20, Q22, Q21   |
| Outer join         | Q13                      |
| Anti-join (NOT IN) | Q16, Q22                 |
| Top-K              | Q3, Q10, Q18, Q21        |
| Window-ish CTE     | Q15                      |
| Big OR-disjunction | Q19                      |

This diversity is intentional — every node-type and every join
algorithm gets exercised at least once in the dataset.

## 5. `scripts/setup_tpch.py` — deep dive

### 5.1 Why DuckDB for data generation?

The official TPC-H generator (`dbgen`) is C code that:

- Doesn't build cleanly on Windows without MSVC + the TPC tools.
- Produces `*.tbl` files that need separate post-processing.
- Has weird flag conventions (`-s 1 -f` for "scale 1, force").

DuckDB ships an **in-memory port** of the same generator as a
loadable extension:

```python
con = duckdb.connect(":memory:")
con.execute("INSTALL tpch;")     # one-time download, cached locally
con.execute("LOAD tpch;")
con.execute(f"CALL dbgen(sf={scale_factor});")
```

This call populates DuckDB's in-memory tables (`region`, `nation`, ...
`lineitem`) with **bit-identical** content to what the C `dbgen` would
produce at the same SF. DuckDB's TPC-H test suite verifies this row-
by-row.

We then export each table to CSV with the canonical `|` delimiter:

```python
con.execute(
    f"COPY {table} TO '{csv_path.as_posix()}' "
    f"(HEADER, DELIMITER '|', FORMAT csv)"
)
```

The `|` delimiter is chosen to match the TPC-H native `.tbl` format —
that way if you ever do want to swap in real `dbgen` output, the load
path is unchanged.

### 5.2 Why `--sf` is a `float`, not an `int`

The TPC-H spec defines fractional scale factors (0.01, 0.1, 0.3, 1, 3,
10, ...). For laptop development SF=0.1 is enormously useful
(~600 K lineitem rows, loads in seconds) while still hitting all 22
query shapes. `int` would block you from that.

### 5.3 Why we generate, *then* load (not stream)

You could in principle stream DuckDB → PostgreSQL row-by-row, but:

- CSV on disk is a **checkpoint** — if the COPY step fails on lineitem
  you don't pay to regenerate.
- It gives you a reproducible artifact you could re-use to seed a
  different test DB (e.g. MySQL, ClickHouse) without re-running
  DuckDB.
- `--skip-gen` lets you iterate on the load step (schema tweaks,
  index changes) in seconds, not minutes.

### 5.4 `copy_expert` + binary file mode

```python
with csv_path.open("rb") as f, conn.cursor() as cur:
    cur.copy_expert(
        f"COPY {table} FROM STDIN WITH (FORMAT csv, HEADER true, DELIMITER '|')",
        f,
    )
```

Critical Windows detail: **open the CSV in binary mode (`"rb"`)**.
PostgreSQL's COPY protocol is byte-exact — if Python opens in text
mode, Windows turns `\r\n` into `\n` on read, which changes byte
counts and confuses COPY. Binary mode bypasses this entirely.

`copy_expert` is also dramatically faster than `executemany` for bulk
loads: it streams data over the COPY wire protocol with no per-row
parsing overhead. SF=1 (~6.5 M rows total) loads in under a minute on
a typical laptop SSD.

### 5.5 Why `ANALYZE` at the end is non-negotiable

PostgreSQL's planner uses `pg_statistic` (column histograms, MCVs,
n_distinct) to estimate row counts. After a bulk COPY, those stats
are stale (or worse, defaulted to the planner's pessimistic guesses).

Without `ANALYZE`, the **first** run of `collect_tpch_plans.py` would
record plans built on garbage statistics, then auto-`ANALYZE` would
fire in the background, and the **second** run would produce
completely different plans — destroying reproducibility.

So we run it explicitly, transactionally, before declaring the
database "ready".

## 6. `scripts/collect_tpch_plans.py` — deep dive

### 6.1 The variant matrix

```python
VARIANTS: dict[str, list[str]] = {
    "default":      [],
    "no_hashjoin":  ["SET enable_hashjoin  = off"],
    "no_mergejoin": ["SET enable_mergejoin = off"],
    "no_nestloop":  ["SET enable_nestloop  = off"],
}
```

This is the *experimental design* for Phase 2A. For each TPC-H
query we collect 4 plans:

1. **`default`** — what PG would actually run in production.
   Ground truth for "best-effort planner".
2. **`no_hashjoin`** — forces PG to use merge / nested-loop joins.
   Often surfaces dramatically slower (or sometimes faster!) plans.
3. **`no_mergejoin`** — forces hash / nested-loop joins.
4. **`no_nestloop`** — forces hash / merge joins. Critical for
   queries where PG defaults to a nested loop on a deceptively small
   estimated cardinality that's actually huge.

**Why this matters for ML:** if every query in your training set is a
`default` plan, your model can only learn "the planner is always
right". With variants, the model sees:

```
(query Q5, plan_A, actual_time = 1.2 s)
(query Q5, plan_B, actual_time = 14.7 s)
(query Q5, plan_C, actual_time = 8.9 s)
```

It now has *contrast* — the raw signal needed to learn which plan
features correlate with low actual time. That's the foundation of
every learned cost model in the literature (Neo, Bao, Balsa, ...).

### 6.2 Why these `SET enable_*` settings, not `pg_hint_plan`

PG's `enable_<method>` GUCs work by adding a **disable_cost** penalty
(default `1.0e10`) to plans that use the disabled method. They do
*not* hard-forbid it — if the only legal plan uses the disabled
method, PG will still pick it.

That's exactly what we want:

- For most queries, the disable forces a different plan we wouldn't
  otherwise see.
- For queries where the disabled method is structurally required
  (e.g. a `WHERE x = (SELECT ... FROM y)` correlation), we still get
  a runnable plan and just record that the variant collapsed back
  to default.
- We don't need a third-party extension (`pg_hint_plan`) — `enable_*`
  is core PostgreSQL.

### 6.3 Session hygiene between measurements

```python
cur.execute("RESET ALL;")
cur.execute(f"SET statement_timeout = {STATEMENT_TIMEOUT_MS};")
for stmt in settings:
    cur.execute(stmt + ";")
```

`RESET ALL` reverts every GUC to its default. Without this, a stale
`enable_hashjoin = off` from the previous variant would silently leak
into the next query's "default" measurement — a classic experiment-
poisoning bug.

We also re-apply `statement_timeout` after `RESET ALL` because the
reset wipes it.

### 6.4 `statement_timeout` — the safety net

```python
STATEMENT_TIMEOUT_MS = 5 * 60 * 1000   # 5 minutes
```

TPC-H Q9 / Q21 / Q22 with `enable_nestloop = off` on SF=1 can degrade
into a plan that takes **30+ minutes**. We'd rather record "timeout
at 5 min" than hang the collector.

When PG cancels the query psycopg2 raises
`psycopg2.errors.QueryCanceled`, which we catch and log without
killing the batch.

### 6.5 Output schema (extends Phase 1 schema)

```json
{
  "query_id":      "q03",
  "variant":       "no_hashjoin",
  "variant_knobs": ["SET enable_hashjoin  = off"],
  "tag":           "join+agg+sort+limit",
  "sql":           "SELECT ...",
  "sql_hash":      "deadbeef",
  "collected_at":  "2026-05-21T..Z",
  "wall_time_ms":  1827.4,
  "summary":       { /* same as Phase 1 */ },
  "plan":          [ /* raw EXPLAIN JSON tree */ ]
}
```

Two new top-level keys vs. Phase 1:

- `variant` — the variant name (string, one of `VARIANTS.keys()`).
- `variant_knobs` — the exact list of GUC overrides applied, captured
  verbatim. Future-proof: if you later add `no_indexscan` you don't
  have to remember which SET statements built that variant.

### 6.6 Filename convention

```
data/tpch/plans/q03__no_hashjoin__a1b2c3d4.json
                ^^^   ^^^^^^^^^^   ^^^^^^^^
                |     |            |
                |     |            SHA-1[:8] of SQL text
                |     |
                |     variant name
                |
                query id
```

Triple-underscore segmentation, easy to `split("__")` in pandas:

```python
df["query_id"], df["variant"], df["sql_hash"] = zip(*df["filename"].str.split("__"))
```

If you re-run the collector after editing Q3's SQL, the hash changes
and you keep both old and new files — same history-preserving pattern
as Phase 1.

### 6.7 The index file

`data/tpch/plans/_index.jsonl` is the same pandas-friendly format as
Phase 1, with an extra `variant` column:

```python
import pandas as pd
df = pd.read_json("data/tpch/plans/_index.jsonl", lines=True)

# How often does the planner pick the same shape across variants?
df.groupby("query_id")["root_node_type"].nunique()

# Which variant actually runs fastest per query?
df.loc[df.groupby("query_id")["execution_time_ms"].idxmin()]
```

This is your Phase-2B starter notebook in two lines.

## 7. Reproducibility checklist

To rebuild the entire TPC-H dataset on a fresh machine:

```powershell
createdb automl_qo
psql -d automl_qo -f db/schema.sql            # Phase 1 tables (optional)
pip install -r requirements.txt
python scripts/setup_tpch.py                  # generates + loads SF=1
python scripts/collect_tpch_plans.py          # 88 plans, ~10-15 min
```

If plans look different from a previous run, suspect (in order):

1. **Scale factor mismatch** — `--sf 1` vs `--sf 0.1` produces wildly
   different plans for the *same query* because cardinalities change.
   Always record SF in your experiment notes.
2. **Different PostgreSQL major version** — planner heuristics evolve
   between PG 14 / 15 / 16 / 17. Pin your PG version per dataset.
3. **GUCs changed** — check `random_page_cost`, `effective_cache_size`,
   `work_mem`, `max_parallel_workers_per_gather`. Defaults vary
   between distros.
4. **`ANALYZE` skipped** — fresh data without stats ⇒ planner uses
   defaults ⇒ wildly pessimistic plans.

## 8. Common pitfalls

- **DuckDB install fails behind a corporate proxy.** The `INSTALL tpch;`
  call downloads the extension from `extensions.duckdb.org`. If your
  network blocks it, set `HTTP_PROXY` / `HTTPS_PROXY` env vars before
  running `setup_tpch.py`.
- **`psycopg2.errors.NumericValueOutOfRange` on COPY.** Usually means
  your locale is interpreting the decimal separator as `,` instead of
  `.`. Set `SET lc_numeric = 'C';` in your PG session, or upgrade
  psycopg2.
- **First TPC-H query is slow on the first run.** PG's buffer cache
  is cold. Run `collect_tpch_plans.py` twice and use the second run
  for any *runtime* analysis; the first run is fine for *plan shape*
  analysis.
- **Setting `enable_nestloop = off` didn't change anything.** Some
  queries (e.g. correlated semi-joins) have no legal alternative.
  This is documented per-query in the `_index.jsonl` — group by
  `(query_id, root_node_type)` to see which variants actually
  produced different plans.
- **Disk space.** SF=1 → ~1 GB of CSV + ~1.5 GB inside PG +
  ~50 MB of plan JSONs. SF=10 multiplies all of these by ~10. Plan
  accordingly.

## 9. What Phase 2B will consume

Phase 2B (feature extraction + cost regression) reads:

- `data/raw/*.json` — Phase 1 synthetic plans (for sanity).
- `data/tpch/plans/*.json` — Phase 2A TPC-H plans (real signal).
- `data/tpch/plans/_index.jsonl` — fast metadata lookup.

The supervised target is unchanged from Phase 1:
**`Actual Total Time × Actual Loops`** at the root (or top-level
`Execution Time`).

What's new: per-`(query_id, variant)` *contrast pairs* let us train
not just a regressor but a **plan-ranking model** — far more useful
for actual plan selection downstream.

## 10. TL;DR

- **`db/tpch_schema.sql`** = spec-faithful TPC-H tables + lightweight
  indexes that preserve planner choice.
- **`db/tpch_queries.sql`** = all 22 reference queries, PG-flavored,
  with `-- @QUERY:` markers for machine parsing.
- **`scripts/setup_tpch.py`** = DuckDB generates the data (no `dbgen`
  build pain), CSV is the checkpoint, psycopg2 `COPY` bulk-loads,
  `ANALYZE` refreshes stats.
- **`scripts/collect_tpch_plans.py`** = 22 queries × 4 optimizer
  variants = 88 plans per run, schema-compatible with Phase 1.
- **The whole point** is variant diversity — without `no_hashjoin` /
  `no_mergejoin` / `no_nestloop` we'd only ever see PG's best guess,
  and a learned model would have nothing to rank against.
